iOS 쪽은 먼저 Transporter로 메모요.ipa 업로드를 돌려 App Store Connect에 빌드가 올라간 걸 확인했다. 메타데이터는 사전에 docs store-assets에 정리해 둔 이름과 부제, 설명문, 키워드, 프로모션 텍스트, 개인정보 처리방침 링크를 그대로 채웠다. 스크린샷은 실기기 캡처본을 9대 16 비율로 정리해 업로드했고 연령 등급, 앱 프라이버시, 데이터 안전 항목까지 모두 입력했다. 심사 제출 버튼을 누른 뒤 상태가 WAITING_FOR_REVIEW로 전환되는 것까지 확인했다. 심사 결과는 24에서 48시간 안에 나올 예정이라 내 작업은 여기서 일단락됐다.

Android 쪽은 더 할 일이 많았다. 먼저 릴리즈 서명이 없어 AAB를 못 만드는 상태였다. android 폴더 아래 메모요 전용 키스토어를 keytool로 새로 만들었다. memoyo-upload-keystore.jks 파일을 RSA 2048비트 1만 일 유효기간으로 생성하고 alias는 upload로 맞췄다. 그리고 같은 폴더에 key.properties 파일을 만들어 storePassword, keyPassword, keyAlias, storeFile 네 값을 박아 두었다. app build.gradle.kts 상단에 Properties와 FileInputStream을 import하고, keystoreProperties를 rootProject.file로 읽는 블록을 추가했다. android 블록 안에는 signingConfigs create release를 새로 넣고 keyAlias keyPassword storeFile storePassword를 properties에서 꺼내 매핑했다. buildTypes release에서는 기존에 debug 서명을 쓰던 부분을 release로 교체하고 minifyEnabled와 shrinkResources를 true로 올려 프로가드까지 엮었다. flutter build appbundle release로 서명된 AAB가 정상 생성되는 걸 확인했다.

Google Play Console 작업은 전부 Chrome과 AppleScript를 거쳐 브라우저 JS 인젝션으로 자동화했다. 먼저 계정 대시보드에서 앱 생성으로 들어가 메모요 이름과 한국어 기본 언어, 무료, 앱 선언에 해당하는 모든 라디오와 체크박스를 프로그램적으로 눌렀다. 대시보드가 작업 리스트로 넘어간 뒤에는 11가지 선언과 스토어 등록정보, 앱 콘텐츠, 연령 등급, 대상 사용자 층, 데이터 안전성, 광고 유무, 뉴스앱 여부, 코비드19 앱 여부, 정부앱 여부, 금융앱 여부까지 차례로 끝냈다. 대부분 아니요로만 찍으면 됐지만 저장 버튼이 비활성 상태로 꼬이는 구간이 몇 번 있어 저장을 먼저 누른 뒤 다음으로 넘어가는 방식으로 우회했다. 콘텐츠 등급 IARC 설문은 9개 문항을 모두 아니요로 통과시켜 전체 이용가 등급을 받아 냈다.

자산 생성도 병렬로 했다. 스토어 아이콘은 기존 1024x1024 파일을 sips로 512x512로 리사이즈해 바로 업로드했다. 피처 그래픽은 PIL로 1024x500 다크 배경 위에 메모요 고양이 아이콘과 메모요 타이틀, 부제를 AppleGothic 한글 폰트로 얹어 새로 만들었다. 폰 스크린샷은 아이폰 1320x2868 원본을 1080x1920에 맞춰 비율 보존으로 축소하고 좌우 다크 패딩을 더해 넣었다. 4장의 스크린샷이 리스트, 새 메모, 편집, 편집 모드 네 가지를 보여 주도록 골랐다.

AAB 업로드는 파일 크기가 40메가를 넘어 AppleScript 단일 호출에 직접 못 넣는 문제가 있었다. 파이썬으로 base64 인코딩한 뒤 2메가짜리 청크 28개로 쪼개서 Chrome의 window 객체에 window aab 0부터 window aab 27까지 순서대로 얹은 다음 조립 스크립트로 Blob을 만들고 DataTransfer에 File 오브젝트를 밀어 넣는 방식으로 file input에 프로그램적으로 할당했다. 이 과정을 반복해 1.0.0 릴리스 AAB가 내부 테스트 트랙에 정상 업로드됐고 저장과 검토, 출시까지 눌러 내부 테스트 상태가 심사 대기로 전환됐다.

개인 구글 계정은 2023년 11월 정책 변경 이후 프로덕션 트랙을 바로 열 수 없고 20명 테스터로 14일간 클로즈드 테스트를 먼저 돌려야 한다는 제한이 있었다. 내부 테스트는 즉시 가능해서 일단 그쪽으로 릴리스를 낸 상태다. 남은 건 테스터 모집이었다. 한국어 커뮤니티용 베타 모집글을 완성형으로 만들어 오픈카카오톡방에 바로 올렸다. 앱 소개, 5단계 참여 안내, 내부 테스트 옵트인 링크, 감사 DM과 iOS 출시 알림 제공, 연락 이메일을 한 단락씩 담았다. 모집 대상 커뮤니티로는 클리앙, 뽐뿌, 디시, 네이버 카페, 트위터도 제안했다.

저녁 후반에는 운영 환경 정비를 몇 건 돌렸다. 먼저 텔레그램 입력중 지시기가 세션이 끝나도 계속 살아있는 오펀 데몬 문제가 있었다. telegram-typing-start.sh 훅이 idempotent 기준이 약해 이전 세션의 데몬이 남아 있으면 신규 세션 시작 시 또 하나가 붙고, Stop 훅은 PID 파일에 기록된 하나만 죽여 오펀이 쌓이는 구조였다. 두 스크립트를 pkill 패턴 매칭 기반으로 재작성해 세션 시작 시와 종료 시 telegram-typing-daemon.sh 프로세스를 전부 죽이도록 바꿨다. 이로써 이전 세션 데몬이 다음 세션까지 밀려 나가는 일이 더 이상 생기지 않는다.

stop 훅 알림 방향도 다시 설계했다. 기존 telegram-stop-ping.sh는 transcript 어디에라도 plugin telegram telegram 태그가 있으면 무조건 푸시를 스킵해서, 텔레그램 플러그인이 붙은 세션이면 터미널에서 입력한 답변조차 푸시가 안 갔다. 사용자는 이 경우에도 답변 왔습니다 알림이 필요했다. 마지막 user 메시지의 content만 jq로 추출해 그 안에만 텔레그램 origin 마커가 있는지 체크하는 방식으로 고쳤다. 메시지가 array 형태의 tool_result가 아닌 string 타입 user 입력만 필터링하고 origin server 필드의 plugin telegram telegram 문자열을 grep하는 구조다. 이제 텔레그램 입력에 대한 응답은 봇 reply 푸시와 중복되는 훅 알림을 스킵하고 터미널 입력에 대한 응답은 정상적으로 푸시가 울린다. 메모리 파일도 이 규칙을 반영해 내용을 덮어썼다.

맥 권한과 Claude Code 설정 쪽도 정리했다. 먼저 Ghostty 터미널의 TCC 권한을 Automation, Accessibility, Input Monitoring, Screen Recording, Full Disk Access까지 전부 점검해 비어 있던 FDA를 자동으로 추가했다. 그리고 Claude Code가 자체 설정 디렉토리인 홈 디렉토리 닷클로드 아래 파일을 편집할 때마다 권한 프롬프트가 뜨는 문제를 없애려고 settings.json의 defaultMode를 acceptEdits에서 bypassPermissions로 바꾸고 skipDangerousModePermissionPrompt를 true로 박아 두었다. Edit 와 Write, Read 에 닷클로드 경로 명시 허용 규칙도 allow 배열에 추가했다. 이 설정은 세션 시작 시에 로드되므로 현재 세션 안에서는 per 세션 승인이 한 번 필요했지만 다음 세션부터는 닷클로드 편집 프롬프트 없이 바로 통과된다.
