2026.04.19 작업일지 v1.0.2

v1.0.1 은 맥 세션만으로 작성된 반쪽짜리였다. 직전까지 Tailscale pubkey 가 안 깔려 있어 rsync 가 막혀 있었고, v1.0.1 을 쓰는 과정에서 그 사실이 드러나 맥 쪽에서 ed25519 키를 새로 만들고 ssh-copy-id 로 윈도우 WSL 에 공개키를 심었다. 그 직후 rsync 가 통과해 오늘자 윈도우 세션 40개가 맥으로 정상 복제됐고, 그 자료를 합쳐 다시 쓴 스냅샷이 이번 v1.0.2 다.

새벽 시간대의 윈도우 WSL 에서는 반대 방향 SSH 기반을 먼저 깔아 두었다. ssh-keygen 으로 ed25519 키를 만들고 tailscale ip -4 로 맥의 Tailnet IP 를 확인한 뒤 ssh-copy-id user at 100.74.85.37 로 맥 방향 공개키를 심었다. 이건 윈도우에서 맥으로 SSH 하기 위한 것이었고, 오후에 맥이 반대 방향을 마저 맞추면서 양방향이 완성됐다. 같은 시간대에 readlink /proc/self/ns/mnt 같은 진단 명령으로 cc 래퍼의 tmux 네임스페이스를 들여다본 흔적도 남아 있다.

02시 부근에는 윈도우 settings.json 에 PreToolUse 훅을 하나 꽂았다. Edit Write MultiEdit 매처로 bash ~/.claude/hooks/auto-allow-claude-dir.sh 를 돌려 .claude 하위 파일 편집 시 반복되던 권한 프롬프트를 줄이는 설정이다. 맥에 이미 들어가 있던 훅을 윈도우에 이식한 형태.

03시 반 부터는 PostToolUse 진행 상황 알림 훅 진단이 본격화됐다. 맥이 오전에 건네 준 7 단계 진단 프롬프트를 따라 설정 상태와 파일 권한과 봇 토큰과 chat-id 캐시를 차례로 체크했다. 그 과정에서 2 분 간격 진행 보고 훅이 다시 살아났지만, 더 큰 문제는 따로 있다는 게 드러났다. 바로 스톱 훅의 flush 경쟁 조건이었다.

04시 언저리 윈도우 터미널에서는 터미널에서 쳐도 텔레그램 답장 오나, 되나, 안녕, 넌 누구니 같은 짧은 입력을 여러 번 넣어 가며 미러링을 실측했다. 긴 응답 자리에 헤더만 오거나 엉뚱한 토막이 붙는 증상이 재현됐고, 이 경험이 쌓이면서 Stop 훅이 턴 종료 직후 파이어되는데 transcript 에 assistant text 가 아직 flush 되지 않은 상태일 수 있다는 가설이 떠올랐다. 윈도우 Claude 가 먼저 telegram-stop-ping.sh 의 last_text 추출 직전에 0.5 초 간격 6 회 재시도 루프를 박아 자체 검증을 마쳤다.

그 보고가 맥 세션으로 건너온 게 13시 20분 경이었다. 맥은 같은 훅의 같은 위치에 동일 루프를 이식했고, ㅎㅎ 와 넌누구냐 를 한 번씩 더 쳐 미러링이 정상 복귀했음을 서로 확인했다. 뒤이어 reference_telegram_stop_hook_race_retry 메모리와 MEMORY 인덱스 한 줄로 이 fix 의 의도와 제거 금지 이유를 박아 두어, 훗날 다른 Claude 세션이 이 코드를 단순화 대상으로 오인해 지우는 일을 막았다.

멀티 기기 운영 규칙도 한 가지 확정했다. /worklog 는 맥에서만 돌린다. 양쪽에서 동시에 돌리면 서로 다른 기준으로 파일명이 밀리고 내용이 충돌할 위험이 있어, 윈도우는 작업만 하고 기록은 맥이 Tailscale rsync 로 양쪽 세션을 당겨와 단일 저자로 모은다. 이 규칙을 세우면서 방금 SSH pubkey 까지 해결되어, 오늘 이후로는 /worklog 한 번이면 맥과 윈도우 양쪽 세션이 자동 통합된다.

남은 작업은 v1.0.1 시점과 같다. 애플 심사 결과 확인, Play Closed Testing Day 5 전 1차 업데이트 aab 업로드, 메모요 iOS UIPasteControl 이식, GitHub CLI 재인증, 미리알림 AppleEvent 권한 정상화가 이어서 처리할 항목이다.
