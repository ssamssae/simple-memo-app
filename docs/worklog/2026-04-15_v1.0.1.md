2026.04.15 작업일지 v1.0.1

오늘 v1.0.0 이후의 시간은 도구 정비와 메모요 출시 준비 두 가지로 크게 나뉜 하루였다. 오전에는 사용량 확인과 이전 날짜 작업일지 소급, 텔레그램 사용성 개선 같은 환경 작업이 있었고, 오후부터는 퇴근 후 시간을 아끼기 위해 메모요 앱스토어 출시를 병렬로 준비했다.

우선 내가 지금 이 순간까지 얼마나 토큰을 쓰고 있는지 한 번에 확인하고 싶어서 usage 라는 이름의 새 스킬을 만들었다. ccusage 도구가 뱉는 5시간 빌링 블록 JSON 을 받아서 현재 블록 진행률과 번 레이트, 블록 종료 시 예상치, 최근 블록 히스토리, 오늘과 이번 주 합계, 모델별 분포를 ASCII 바 그래프로 출력하는 구조로 짰다. 주간 구독 한도 퍼센트는 Anthropic 이 공개 API 를 제공하지 않아 직접 못 뽑고, 대신 관찰된 최대 블록을 기준으로 하는 상대 사용률로 보여 준다. 앞으로 터미널에서도 텔레그램에서도 usage 한 번이면 한눈에 현황 파악이 가능하다.

4월 11일부터 13일까지는 그동안 작업일지가 비어 있었다. 세 날짜를 모두 소급해서 작성했다. 11일은 메모요 앱이 태어난 날이고 개인 홈페이지 daejong-page 초판도 이날 올라갔다. 본문만 남기는 메모 구조, 스와이프로 즐겨찾기와 삭제를 붙이는 UX, 드래그 핸들로 순서 바꾸기, 다크 테마와 아이콘 반복 수정, iOS 재실행 크래시 디버깅까지 하루 안에 이어진 긴 흐름을 정리했다. 12일은 앱 네 개가 병렬로 돌아간 가장 묵직한 날이었다. 메모요 iOS UIScene 크래시 수정과 더치페이 계산기 토스 스타일 키패드 신규 제작, 약먹자 재설계, 고독사 방지앱 초기 세팅이 한 문서에 들어갔다. 13일은 약먹자 초기 커밋이 만들어진 날이고 텔레그램과 Claude Code Channels 에 대한 리서치가 저녁 시간 대부분을 차지했다. 세 문서 모두 산문체로 작성해 simple_memo_app 저장소의 docs worklog 폴더에 커밋 푸시했고, daejong-page 쪽 worklog 폴더와 index.json 에도 동시 반영했다.

텔레그램 사용성 문제도 몇 가지 손봤다. 세션이 답변을 준비하는 동안 입력중 표시가 금방 사라져서 내가 멈춘 건지 작동 중인지 구분이 안 된다는 피드백이 있었다. 처음에는 PreToolUse 훅에서 sendChatAction 을 4초 디바운스로 쏘는 방식을 만들었다가, 생각만 하는 구간에서는 툴이 안 돌아가 훅도 안 터진다는 한계가 있어 방식을 바꿨다. 세션 시작 시점에 백그라운드 데몬을 떠서 4초마다 타이핑 신호를 계속 쏘고 응답이 끝나는 Stop 훅에서 바로 죽이는 구조로 재설계했다. UserPromptSubmit 훅이 데몬을 스폰하고 Stop 훅이 PID 파일 기반으로 종료한다. 이 세션 안에서는 훅이 반영되지 않지만 다음 재시작부터는 답변이 끝날 때까지 내내 입력중 이 유지된다.

코워크 쪽에서 스케줄러가 안 도는 문제도 한 차례 진단했다. 원래 매일 아침 9시에 브리핑 봇으로 사이드 프로젝트 아이디어를 보내기로 해놓았는데 오늘 한 건도 도착하지 않았다는 제보였다. 로컬 기준 확인 가능한 모든 스케줄 저장소를 뒤졌다. Claude Code 내장 CronList, claude.ai 의 v1 code triggers 엔드포인트, macOS 사용자 crontab, launchd, Claude 데스크탑 앱의 내부 저장소까지 전부 0건이었다. 그런데 사용자 쪽 말로는 Claude Cowork UI 에 네 개가 등록돼 있다고 했다. 즉 Cowork 의 스케줄러는 Claude Code CLI 가 보는 트리거 저장소와 다른 곳에 있고, 내가 직접 조회하지 못한다는 결론이 나왔다. 추가로 Documents Claude Scheduled 폴더 아래 daily side project briefing 이라는 스킬 정의 파일을 찾았다. 그 스킬은 outputs 폴더에 마크다운 파일을 저장하는 것까지만 작성돼 있고 텔레그램 봇으로 전송하는 단계가 아예 들어 있지 않았다. 결국 Cowork 에 스케줄은 있어도 실행 결과가 텔레그램으로 나갈 경로 자체가 없었고, 이 네 스케줄을 Claude Code 세션에서 정상 작동하도록 다시 엮는 작업을 오늘 할일에 넣어 두었다.

오후부터는 출시 전 단계를 확인하면서 구글과 애플 양쪽 계정 상태를 정리했다. 구글 플레이 콘솔은 개인정보 처리 요구 사항이 더 까다로워져 전화번호 인증과 신원 확인 같은 절차가 순차로 열리는 구조였다. 처음엔 사용자가 단순히 심사가 늦는다고 느꼈지만 실제로는 본인이 더 채워야 할 단계가 남아 있어서 공이 사용자 쪽에 있는 상태였다. 계정 세부정보 페이지에서 전화번호 인증을 진행해 확인됨 표시까지 받아 뒀다. 애플 쪽은 더 앞서 있었다. App Store Connect 나의 앱 페이지에 메모요 약먹자 더치페이 계산기 세 앱이 이미 등록돼 iOS 1.0 제출 준비 중 상태로 들어 있었다. 개발자 계정이 활성화돼 있다는 뜻이어서 그 자체로 안심할 수 있었고 남은 건 각 앱의 메타데이터와 빌드 업로드 정도였다.

이어서 메모요 출시 준비를 한 번에 여러 갈래로 돌리는 병렬 작업에 들어갔다. 먼저 Android 쪽은 applicationId 가 Flutter 기본값인 com.example.simple_memo_app 으로 남아 있어서 Google Play 업로드 자체가 막히는 상태였다. android app build.gradle.kts 의 namespace 와 applicationId 를 com.daejongkang.simple_memo_app 으로 바꾸고 Kotlin 패키지 폴더를 com example 에서 com daejongkang 으로 옮기고 MainActivity.kt 의 패키지 선언을 수정했다. linux CMakeLists 의 APPLICATION_ID 도 함께 맞췄고 flutter clean 과 pub get 을 돌린 뒤 flutter build apk release 로 정상 APK 가 떨어지는 걸 확인했다.

iOS 쪽 붙여넣기 프롬프트 제거 작업도 같이 했다. 매번 다른 앱에서 복사한 텍스트를 메모요 텍스트필드에 붙여넣을 때마다 다른 앱에서 붙여넣기 시스템 프롬프트가 뜨는 게 거슬린다는 요청이었다. iOS 16 이상에서만 가능한 UIPasteControl 을 네이티브로 이식하는 방향으로 잡고 ios Runner 폴더에 PasteButtonPlatformView 라는 Swift 파일을 새로 만들었다. PlatformViewFactory 로 등록되는 UIView 하나가 UIPasteControl 을 자식 뷰로 품고, 탭되면 pasteItemProviders 콜백에서 NSItemProvider 로부터 텍스트를 꺼내 MethodChannel 로 Dart 쪽에 전달하는 구조다. AppDelegate 에 factory 등록 코드를 넣고 Runner 의 project.pbxproj 에 Swift 파일을 네 군데 ( PBXBuildFile, PBXFileReference, PBXGroup children, PBXSourcesBuildPhase files ) 에 모두 등록했다. Dart 쪽에는 lib widgets paste_button.dart 라는 위젯을 새로 만들어 UiKitView 로 PlatformView 를 감싸고 onPaste 콜백을 상위 화면에 연결했다. 메모 편집 화면은 이 위젯을 AppBar 오른쪽 저장 버튼 왼편에 배치하고, 커서 위치 또는 선택 영역을 텍스트로 치환하는 insertPastedText 로직을 추가했다. flutter build ios release 로 컴파일까지 정상 완료되는 걸 확인했다. 실제 프롬프트 제거 동작은 실기기에서 다른 앱의 텍스트를 복사한 뒤 이 버튼을 눌러 확인해야 한다.

스토어 제출에 필요한 메타데이터와 자산도 정리했다. docs store-assets 폴더를 새로 만들고 metadata.md 문서에 앱 이름 후보와 부제 세 가지, 카테고리 생산성, 연령 등급 4 플러스, 한국어 설명문과 영문 참고 번역, 프로모션 텍스트, 키워드 100자 내 리스트, 짧은 설명, App Privacy 와 데이터 안전 섹션 답변까지 차례로 박아 두었다. 1024x1024 메모요 아이콘은 기존 iOS Icon-App-1024x1024 파일을 파이썬 PIL 로 열어 RGB 만 남기고 알파 채널이 없는 PNG 로 저장했다. 메모요 개인정보 처리방침도 daejong-page 저장소에 privacy.html 로 새로 추가했다. 앱이 어떤 데이터도 수집하지 않고 로컬에만 저장한다는 점을 한국어로 풀어 쓰고 연락처와 홈페이지 주소까지 포함했다. 이 페이지는 ssamssae.github.io daejong-page privacy.html 에서 바로 열람된다.

마지막으로 실제 릴리즈 아카이브와 IPA 까지 뽑았다. flutter build ipa release export-method app-store 명령으로 개발팀 자동 서명과 함께 빌드를 돌렸고 Xcode archive 성공, 버전 1.0.0 빌드 5, 표시 이름 메모요, 번들 아이디 com.daejongkang.simpleMemoApp 로 깨끗하게 패키징됐다. 결과 파일은 build ios ipa 폴더의 메모요.ipa 이고 크기는 18메가바이트다. 업로드는 Transporter 앱에 드래그드롭 하거나 xcrun altool 로 CLI 업로드 두 가지 경로가 안내됐다. 남은 건 실기기 스크린샷 세 장 이상 준비와 Transporter 업로드뿐이다.

이 흐름과 별개로 오늘 할일도 꾸준히 늘었다. usage 관련 퇴근 후 작업, Cowork 스케줄러 네 개 정상 작동 수정, 메모요 Google Play Store 출시, 메모요 Apple App Store 출시, 메모요 iOS UIPasteControl 이식, daejong-page 포트폴리오 웹 프로젝트 추가, daejong-page 버전업까지 총 일곱 건이 진행중 섹션에 쌓였다. 이 중 UIPasteControl 이식은 오늘 실제로 코드와 빌드까지 완료했지만 실기기 검증이 남아 있어 완료 처리는 유보 상태다.

남은 작업은 퇴근 후 메모요 실기기 스크린샷 촬영과 Transporter 를 이용한 IPA 업로드, App Store Connect 에서 메타데이터 채우기와 심사 제출, Google Play Store 쪽은 신원 확인 남은 절차까지 마무리하고 나서 새로 빌드한 APK 를 업로드하는 단계, 그리고 Playwright MCP 를 Claude Code 에 연결해 주간 사용량 퍼센트를 실제로 파싱하는 일이다.
