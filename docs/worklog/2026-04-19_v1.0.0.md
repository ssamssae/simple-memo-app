2026.04.19 작업일지 v1.0.0

오늘은 오전에 메모요 Android 선택 박스 2톤 이슈와 상단 버튼 정리, APK 16.7MB 최적화를 끝낸 뒤 기존 2026-04-19.md 로 1차 기록을 남겼고, 점심 직전에는 멀티 기기 운영 중 발생한 tmux 사고를 다루며 운영 규칙을 정비했다. 이번 v1.0.0 스냅샷은 그 사고 분석과 후속 정비가 중심이다.

사고는 맥이 아닌 다른 기기에서 Claude Code 세션을 /exit 명령으로 빠져나간 순간 시작됐다. 그 기기의 tmux 자체가 함께 꺼져버리면서 다시 cc 로 들어가려 해도 붙지 않았고, 텔레그램 채널도 같이 끊겨 보였다. 상황이 이상해 맥 본진부터 점검했는데 맥 쪽은 문제없이 계속 살아있었다. tmux 의 claude 마스터 세션이 오전 11시 55분 KST 부터 중단 없이 돌고 있었고, 텔레그램 MCP 플러그인도 연결 상태였으며 morning-briefing todo-reminder review-status-check 같은 launchd 잡도 전부 로드돼 있었다. 다른 기기에서 보기에는 끊긴 것 같아도 맥 쪽 기준으로는 아무 이상이 없었다는 뜻이다.

원인을 파고들어 보니 cc 래퍼의 tmux 공유 구조가 핵심이었다. ~/bin/cc 는 claude 라는 마스터 세션을 detached 상태로 한 번만 만들고, 각 터미널이 cc 를 실행할 때마다 claude-PID 형태의 뷰어 세션을 새로 만들어 마스터 그룹에 붙이는 방식이다. 뷰어에는 client-detached 훅으로 kill-session 이 걸려 있어서 뷰어가 끊기면 그 뷰어만 자동 정리된다. 이 구조 자체는 Ghostty 와 iPhone Termius 가 각자 터미널 사이즈를 유지하면서 같은 Claude 프로세스를 공유하게 해 주는 영리한 설계지만, 원격 기기에서 /exit 을 치면 문제가 된다. /exit 은 뷰어 detach 가 아니라 claude 프로세스 자체의 종료 명령이기 때문에, 마스터 세션의 유일한 구동 명령이 죽고 그 여파로 마스터 tmux 전체가 붕괴해 다른 모든 뷰어들까지 한꺼번에 튕긴다.

그래서 원격 기기의 올바른 종료 방법을 세 가지로 정리해 두었다. /clear 로 컨텍스트만 비우고 세션은 유지하는 방법, Ctrl b d 로 tmux detach 만 하는 방법, 그냥 터미널 창을 닫아 뷰어만 자동 정리되게 하는 방법이다. /exit 은 맥 본진에서 진짜로 세션을 끌 때에만 쓰는 것으로 기준을 잡았다. 이 운영 규칙은 feedback_cc_multi_device_exit.md 메모리로 저장하고 MEMORY.md 인덱스에 한 줄로 등록해 두어, 다음에 같은 상황이 생기면 Claude 가 자동으로 이 규칙을 참고하도록 만들었다.

사고 복구 과정에서 사용자가 맥 Ghostty 터미널에 cc 를 쳤을 때 exited 문자열만 뜨고 프롬프트로 돌아오는 현상도 확인했다. tmux ls 로 다시 확인했더니 마스터 claude 세션과 여러 뷰어 세션이 모두 살아 있었기 때문에, exited 는 해당 뷰어가 detach 될 때 뜨는 표시일 뿐 마스터와는 무관하다는 것도 같이 정리했다. 마스터가 살아 있는 한 cc 를 한 번 더 치면 바로 재접속된다는 점도 메시지로 전달했다.

폰 Termius 를 종료했을 때 Ghostty 가 함께 꺼지는 것처럼 보이는 보고도 들어왔는데 tmux 쪽 상태는 그대로였다. 각 터미널이 claude-PID 별 별개 뷰어를 갖고 client-detached 훅도 자기 것만 대상이기 때문에 이론상 한쪽 detach 가 다른쪽을 끌 이유가 없다. 실제 창이 완전히 닫혔다면 Ghostty 자체의 command 종료 시 창 닫기 설정이거나 마지막 탭 정책 같은 앱 레벨 이슈일 가능성이 커서, 현장 확인을 요청하는 쪽으로 남겨 두었다.

데스크탑 쪽 인프라 작업도 한 건 정리했다. 장시간 작업 중 2분마다 텔레그램으로 진행 상황을 보내 주는 PostToolUse 훅이 데스크탑에서는 작동하지 않는다고 해서, 데스크탑 Claude Code 창에 그대로 붙여넣을 수 있는 진단 프롬프트를 만들었다. ~/.claude/settings.json 에 훅 등록 상태 확인, ~/.claude/hooks/telegram-progress-report.sh 파일 존재와 실행 권한, ~/.claude/channels/telegram/.env 의 봇 토큰이 데스크탑용 @Myclaude2 인지 확인, /tmp 아래 chat-id 캐시와 last-report 타임스탬프 확인, 훅 수동 실행과 jq 설치 확인까지 7단계로 구성했다. 맥의 봇 토큰을 데스크탑에 그대로 쓰면 409 Conflict 로 양쪽이 충돌한다는 점을 명시해 두었다.

남은 작업은 오전 기록과 같다. 애플 심사 결과 확인과 Play Closed Testing Day 5 이전 1차 업데이트 aab 업로드, 메모요 iOS UIPasteControl 이식이 이어서 처리할 항목이다. 데스크탑 쪽 훅 진단 결과는 사용자가 해당 기기에서 프롬프트를 돌려 본 뒤 결과를 들고 오면 그때 이어서 수정할 예정이다.
