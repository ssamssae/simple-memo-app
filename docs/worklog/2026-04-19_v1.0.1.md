2026.04.19 작업일지 v1.0.1

v1.0.0 스냅샷 이후 오후에는 텔레그램 스톱 훅 미러링 버그를 잡는 작업이 이어졌다. 사용자가 데스크탑 터미널에 뭔가를 입력했을 때 @Myclaude2 봇으로 오는 응답이 터미널 화면과 다르다는 보고가 시작점이었다. 공유된 스크린샷에는 @MyClaude 와 @Myclaude2 두 채팅창이 나란히 있었고 양쪽 모두 짧은 인사에는 제대로 반응하다가 긴 응답 자리에 헤더만 덩그러니 오거나 엉뚱하게 짧은 메시지가 얹혀 있는 현상이 섞여 있었다. @MyClaude 쪽도 같은 증상이라는 너도그래 한 마디로 맥도 예외가 아님이 확인됐다.

초기 추정은 틀렸다. reply 툴과 스톱 훅 미러 경로의 이중 발사를 의심해, 텔레그램 인바운드에 대해 reply 툴로 풀내용을 보내고 그와 별개로 스톱 훅이 터미널 응답을 또 미러해서 이중으로 쌓이는 구조일 것이라는 가설을 세웠다. 하지만 실제 transcript jsonl 을 직접 조회해 보니 단순한 텍스트 응답은 제대로 추출되고 있었고 길이 집계도 예상 범위였다. 이 가설로는 보이는 증상이 설명되지 않았다.

진짜 원인은 사용자가 데스크탑에서 먼저 찾아 알려 주었다. 스톱 훅은 턴이 종료되는 순간 파이어되는데 transcript jsonl 파일에는 어시스턴트의 마지막 텍스트 블록이 아직 flush 되지 않은 상태일 수 있다. 그 찰나에 jq 가 파일을 읽으면 빈 문자열이나 한 단계 전의 텍스트만 잡혀 나가고, 그래서 텔레그램에는 헤더만 도착하거나 짧은 토막이 붙는 식이 된다. 비동기 파일 쓰기와 훅 실행 사이의 전형적인 경쟁 조건이었다.

수정은 짧고 분명했다. last_text 추출 직전에 0.5 초 간격으로 여섯 번 재시도하는 루프를 넣어 최대 3 초까지 기다리도록 했다. 대부분은 처음 한두 시도 안에 flush 가 잡히지만 여유 폭을 두었고, 그래도 비어 있으면 원래대로 skip 하도록 남겨 무한 폴링은 피했다. 데스크탑 훅은 사용자가 먼저 고쳐 두었고 맥 훅 telegram-stop-ping.sh 의 last_text 블록에도 동일 루프를 이식했다.

검증은 맥 터미널에서 ㅎㅎ 와 넌누구냐 를 쳐 보는 식으로 이뤄졌다. 각각의 짧은 응답이 터미널 응답 헤더와 함께 @MyClaude 로 제대로 도착했고, 사용자 쪽에서도 잘된다 한 마디로 확인했다. race 가 실제 재현되던 환경에서 재시도 루프가 빈 미러링을 막아 준다는 점을 서로 눈으로 확인한 셈이다.

후속으로 참조 메모 한 건을 새로 저장했다. reference_telegram_stop_hook_race_retry 라는 이름으로 이 루프의 의도와 제거 금지 이유를 남겨, 훗날 다른 Claude 세션이 이 코드를 보고 단순화 대상으로 착각해 지우지 않도록 기준을 박아 두었다. MEMORY 인덱스에도 한 줄 추가했다.

작업일지 운영 쪽으로도 한 가지 규칙을 확실히 해 두었다. 윈도우 데스크탑 세션에서는 /worklog 를 따로 돌리지 않는다. 맥이 Tailscale rsync 로 데스크탑 세션을 끌어와 통합 일지를 쓰는 구조이기 때문에 양쪽에서 동시에 /worklog 를 실행하면 서로 다른 기준으로 파일명이 밀리고 내용이 충돌할 수 있다. 데스크탑은 순수 작업자 역할에 머무르고 기록은 맥이 단일 저자로 모은다.
